# Track 3 — open-opticon as the minimal open attestable sensor (software)

This maps open-opticon onto **Track 3** of the credible-sensors program (a small
open device — secure microcontroller + microphone — that captures locally and
emits only a verifiable result). open-opticon is the **software** side of that
track; the physical sensor, secure-element root, Pi target, and enclosure are the
hardware side. The two meet at one documented contract: a **P-256 signature over
a canonical CBOR payload** (`src/common/he_payload.h`).

> Scope discipline, up front: this document separates what is **proven on QEMU**
> (Tier 1, shared embedded test key) from what **needs hardware** from what is
> **frontier**. Nothing hardware-tier is claimed as proven. See
> [THREAT_MODEL.md](THREAT_MODEL.md) for the full honest scope.

## What it is

A microphone whose capture and event detection run inside an Arm TrustZone secure
world (OP-TEE). The raw audio is zeroized in-enclave and the only thing that
leaves the device is a **signed, non-reconstructable predicate** — "alarm tone
detected", "voice present", "occupied" — bound to the same key whose firmware
identity remote attestation proves. Per challenge the in-TEE `ATTEST_AUDIO`
command runs the detector, `TEE_MemFill(pcm, 0)` zeroizes the audio, then it
encodes and signs the canonical payload (`src/optee/ta/he_audio_ta.c`). This
inverts the signed-camera model: instead of signing raw content after capture, it
attests the honest transformation inside the enclave and emits only the bound
minimal result.

## The two properties

The program asks for two properties, verifiable not trusted: **Integrity** and
**Access**.

**Integrity is directly proven.** A bystander scans the QR; a fresh nonce is
issued; the device returns a bundle the open verifier checks against five gates
(`src/verifier/bound.go`):

- Gate 1 — ECDSA-P256 signature over SHA-256(payload) under the attested key.
- Gate 2 — freshness: the signed nonce equals the issued challenge (a photographed
  or replayed QR fails).
- Gate 3 — anti-replay: the monotonic counter strictly advances.
- Gate 4 (optional) — stream continuity: `prev_digest` chains onto the prior
  window, so a *suppressed* window is a detectable gap.
- Gate 0 (optional) — endorsement pin: the key matches the enrolled device.

In parallel, the OP-TEE attestation client + Veraison verify firmware identity for
the same nonce. Together: genuine published firmware produced this exact output,
now.

**Access is enforced structurally, not by policy.** The only code path that emits
anything emits a count/event predicate — never speech — and the firmware is public
and measured, so "who can access the voice" has a verifiable answer: no one,
including the operator, because raw audio is zeroized in-enclave and there is no
source path that emits it. The non-panopticon guarantee is "only as strong as the
public audit of the firmware source": attestation makes the promise checkable; a
human still reads the code.

## The five questions

The walk-up verifier page (`/v`, served by `he-challenge`) answers the program's
five questions in plain language on a PASS, each with a "show me the proof" panel
that reveals the literal verified artifact and states what it does **not** prove.

| Question | How opticon answers it | Tier |
|---|---|---|
| (1) What is captured | Only a predicate leaves; raw PCM is zeroized in-enclave. `config_hash` binds which detection policy ran. | output: proven · in-enclave wipe: firmware/attestation |
| (2) Where it goes | Nowhere as audio. The wire output is the bound CBOR predicate + signature; the audio never leaves. | proven |
| (3) Who accesses / releases | No retained data to release; integrity (who can mint a verdict) is what is enforced. Device identity needs a hardware key. | integrity only · device-id needs hardware |
| (4) How long kept | Audio lives in the enclave for milliseconds, then `TEE_MemFill(…,0)`. Only a monotonic counter persists — no audio. | wipe: firmware/attestation |
| (5) How used | A coarse verdict under a published policy; `config_hash` makes the rules auditable from source. | proven (policy bound) · model quality: frontier |

The walk-up page is deliberate about legibility (the program's measured concern
that people misread status lights): one binary verdict, words carrying the meaning
not color alone, and a visible evidence tier on every answer so a green screen
never implies more than the signature carries.

## Honest scope

**Proven on QEMU / any Linux box (Tier 1, shared embedded test key).** The detector
classification, the deterministic-CBOR payload golden vector, and the full crypto
+ binding + verification pipeline are tested with the exact P-256 test key and
SHA-256→ECDSA the QEMU TA uses. The OP-TEE path has been run end-to-end on QEMU to
a green, `affirming` Veraison attestation. The verifier rejects tampered payloads,
stale nonces, replayed counters, substituted keys, pin mismatches, and
non-canonical CBOR. Tier 1 proves *genuine published code, fresh, output bound* —
**not** device identity, because the key is shared.

**Needs hardware (speced and reviewed, not proven).** Device identity / anti-clone
(a non-extractable key: i.MX CAAM, or the ST secure element on the T3 Pi build —
see [PI_ST_ELEMENT.md](PI_ST_ELEMENT.md)); rollback-proof anti-replay (a hardware
monotonic counter / RPMB); physical tamper response (the enclosure + key-destroy,
explicitly theatre-grade until a real anti-tamper mesh / secure-element zeroization
lands); secure capture (moving PCM into secure-world I2S/PDM so audio never enters
the normal world). Two integration requirements the guarantee depends on: the raw
`SIGN_DATA` PTA must be restricted to the audio TA, and the tamper gate must cover
every key-using command.

**Frontier (open problems, not laptop tasks).** Replacing the heuristic detector
with a small *independently-audited* learned model (the word "audited" is
load-bearing); a public manufacturer endorsement root replacing self-provisioning;
on-chain domain separation against cross-instance replay; PIR/FHE where TEE
side-channel resistance is insufficient.

**Headline scope:** opticon proves software integrity + output provenance, not
nation-state confidentiality.

## Why opticon already fits a heterogeneous hardware root

T3 expects an ST secure element on a Raspberry Pi. opticon is built for it:

- The detector and host code are integer-only C that compiles natively on
  Raspberry Pi OS; OP-TEE attestation targets the RPi 3B+ as the Tier-1-on-silicon
  step (proven on QEMU today; the Pi run is needs-hardware); the Go verifier
  cross-compiles to the Pi with CGO off and no toolchain.
- The verifier is **root-agnostic**: it checks a P-256 signature against an
  enrolled public key (`src/verifier/bound.go`) and does
  not care which silicon produced it. An ST secure-element key drops in exactly
  where the i.MX CAAM key or the QEMU embedded key sits — only the public key is
  enrolled, no verifier change.
- This generalizes to the k-of-n quorum (`src/verifier/quorum.go`): a verdict is
  trusted only if k of n *enrolled* independent roots verify and agree, matched to
  a root by signing key. Heterogeneous roots — an OP-TEE/ST-rooted device, a
  non-TEE ZK leg, a TPM quote — are the design, not a retrofit. (The quorum
  verifier is built and tested; enrolling genuinely heterogeneous *silicon* as a
  second live leg remains integration work.)
