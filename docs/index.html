How it works

One chain, three independent links

Break any link and the verdict turns red.

01 / attest

Prove the firmware

The enclave emits a PSA attestation token. Veraison confirms the running code is the exact published firmware — not a tampered build.

02 / bind

Bind the verdict

That same attested key signs a minimal CBOR predicate bound to a verifier-chosen nonce and a monotonic counter.

03 / verify

Verify the output

Anyone checks the signature, the firmware identity, the freshness, and the anti-replay counter. No audio. No trust required.

Non-panopticon by construction

The only thing that can leave the enclave is a ~95-byte verdict. There is no code path that exports samples — and the attestation proves it is that code path.

Root of capture, not root of trust

You verify a proof rather than trusting an operator or vendor. The TEE is the root of capture; the math is the root of trust.

Fresh every time

A replayed or pre-recorded "all clear" is rejected: each verdict is bound to a fresh nonce and a counter that must strictly advance.

Hardness tiers

Same verifier, three levels of guarantee

Higher tiers add hardware. Items marked proven today ran on a laptop with no special hardware.

Tier 1 — genuine code proven

  • OP-TEE on Arm TrustZone (QEMU); Veraison affirming
  • In-enclave Goertzel detector; audio zeroized in the TA
  • Output signed by the attested key; verify PASS, tamper FAIL

Tier 2 — device identity needs i.MX CAAM

  • proven verifier-side pinning: wrong key → FAIL, genuine → PASS
  • he_host --key-hex path wired end-to-end
  • Hardware adds only non-extractability (CAAM black key)

Tier 3 — tamper enclosure fail-closed proven

  • GPIO loop: breach → key erased + TA flag latched
  • In QEMU: after --trip, attestation returns TEE_ERROR_SECURITY
  • Production: route the loop into the secure element / CAAM

Design

Why a TEE, and not ZK or FHE

For a real-time sensor that captures the physical world, a TEE is the only approach that is fast, cheap, and end-to-end today — and it composes with the others later.

ApproachReal-time on a $5–60 boardHides raw audioProves which code ran
TEE + attestationYes, integer detector in-enclaveYes, zeroized in the TAYes — that is what attestation proveschosen
ZK proof of inferenceNot for live audio yetYesYes, of the computationfuture
FHENo, too slow to streamYesNo, not by itselfnot viable now
A TEE is a hardware trust assumption, not a math proof. The roadmap hardens it: reproducible builds (anyone recomputes the measurement), a 2-of-3 multi-prover so no single TEE vendor is load-bearing, and eventually a ZK proof of the attestation. See WHY_TEE.md and THREAT_MODEL.md.
