01What it actually proves

Three links in one chain. Break any link and the verdict turns red.

attest the firmwareThe enclave emits a PSA attestation token. Veraison confirms the running code is the exact published firmware — not a tampered build.
→
bind the verdictThat same attested key signs a minimal CBOR predicate (alarm_tone/voice/none) bound to a fresh nonce + monotonic counter.
→
verify the outputAnyone checks the signature, the firmware identity, the freshness, and the anti-replay counter. No audio. No trust.

Non-panopticon by construction

The only thing that can leave the enclave is a ~95-byte verdict. There is no code path that exports samples — and the attestation proves it's that code path.

Root of capture, not root of trust

You're not asked to trust the operator or the vendor. You verify a proof. The TEE is the root of capture; the math is the root of trust.

Fresh every time

A replayed or pre-recorded "all clear" is rejected: each verdict is bound to a verifier-chosen nonce and a counter that must strictly advance.

03Three tiers of hardness — honest status

The same verifier logic across all three. Higher tiers add hardware guarantees; what's marked proven today ran on this laptop with no special hardware.

Tier 1 · genuine code proven today

Prove the running enclave is the exact published firmware, producing a minimal verdict for a fresh challenge.

  • OP-TEE on Arm TrustZone (QEMU), Veraison verdict affirming
  • In-enclave Goertzel detector; audio zeroized in the TA
  • Bound output signed by the attested key; host verify PASS / tamper FAIL

Tier 2 · device identity needs i.MX CAAM

Bind the signing key to one physical device with a non-extractable hardware key, so a clone can't impersonate it.

  • proven today verifier-side pinning: clone (wrong key) → FAIL, genuine → PASS
  • he_host --key-hex <FullKey> path wired end-to-end
  • Hardware adds only non-extractability (CAAM black key) — same gate

Tier 3 · tamper enclosure fail-closed proven

Open the see-through mesh case → the key dies and the enclave refuses to attest, even with correct firmware.

  • GPIO loop watcher: breach → key erased + TA flag latched
  • In QEMU: after --trip, attestation returns TEE_ERROR_SECURITY
  • Production: route the loop into the secure element / CAAM mesh

04Why a TEE — and not ZK or FHE?

We considered the alternatives. For a real-time sensor that has to capture the physical world, the TEE is the only approach that's fast, cheap, and end-to-end today — and it composes with the others later.

approachreal-time on a $5–60 boardhides the raw audioproves which code ranverdict
TEE + attestationyes, integer detector in-enclaveyes, zeroized in the TAyes — that is what attestation proveschosen
ZK proof of inferencenot for live audio yet (orders of magnitude too slow)yesyes, of the computationfuture: prove the TEE's claim
FHEno, far too slow for streamingyesno, not by itselfnot viable now

The honest framing: a TEE is a hardware trust assumption, not a math proof. So the roadmap hardens it — reproducible builds (anyone can recompute the measurement), a 2-of-3 multi-prover so no single TEE vendor is load-bearing, and eventually a ZK proof of the attestation. Full reasoning in WHY_TEE.md and THREAT_MODEL.md.
